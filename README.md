## Design Report

### OOP

\*\***Delegation**:\*\* `BattleScreen.java` lines 17-18 and 94-96 use delegation through the 'UI' class. 'BattleScreen' stores a 'UI' object and calls 'ui.draw(lives, wave, score)' instead of directly drawing every text label, life icon, and score display itself. This keeps the battle logic separate from display formatting.



\*\*Method **overriding**:\*\* 'Screen.java' lines 7-16 defines an abstract superclass with abstract 'update()' and 'draw()' methods. 'BattleScreen.java' lines 38-75 and 'PauseScreen.java' lines 16-24 override these methods to provide different behaviour for each screen. This allows 'ShadowAliens.java' line 52 to call 'currentScreen.update(input)' without needing to know which specific screen is currently active.



### Design choice

\*\*Design choice:\*\* The most impactful design choice was moving UI rendering into a separate 'UI' class instead of keeping all drawing code inside 'BattleScreen'. 'BattleScreen' already controls player movement, enemy updates, projectile updates, collision detection, object removal, scoring, and time scale. If the UI drawing code had stayed there as well, the class would be much longer and harder to read.



The implemented design gives 'UI' responsibility for drawing the wave, score, player lives, pause title, controls list, and time scale. For example, 'UI.java' lines 13-40 store the UI-related fields, while lines 88-113 perform the drawing. Many of these fields are private, which also supports **encapsulation** by keeping UI layout details inside the 'UI' class.



An alternative design would be to draw all UI elements directly in 'BattleScreen.draw()'. That would work, but it would mix game state logic with formatting and text positioning. The current design is better because 'BattleScreen' can focus on gameplay control, while 'UI' focuses only on presentation.
